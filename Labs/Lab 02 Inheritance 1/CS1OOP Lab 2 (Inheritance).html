3. Reuse by inheritance: Shape hierarchy

Look at the classes in the uk.ac.aston.oop.inheritance.shapes package. Compare Circle, Ellipse and Rectangle. Do you see the similarities?

They all have getX, getY, getWidth, getHeight and draw. There is too much repeated code here. We are going to use inheritance to avoid this repetition.

Creating the base Shape class

We want to extract all the common parts of these classes to a Shape superclass.

Step 1. Create a Shape class inside the uk.ac.aston.oop.inheritance.shapes package, by right-clicking on the package and selecting New - Class. Enter "Shape" in the "Name" field and press "Finish". You will find that Eclipse has created the file with this starting code for you:

package uk.ac.aston.oop.inheritance.shapes;

public class Shape {

}

Step 2. If you look at Rectangle and Ellipse, you will see that they store the position of the upper left corner, and their width and height. This is more flexible than Circle (which just stores the center and the radius), so we will use their approach to represent shapes. Add private double fields to Shape to store the upper left corner's X position, the upper left corner's Y position, the width, and the height.

Step 3. Add a constructor that takes four double parameters and stores them into the fields you added in the previous step. Remember that a public constructor for a type Type is declared this way:

public Type(parameters) {
  body
}

Step 4. Add the appropriate getter methods for the fields: getX(), getY(), getWidth(), and getHeight(). These methods should just return the value of those fields: you need them because the fields are private (to help with encapsulation and implementation hiding) and they will not be accessible from outside the class otherwise.

Step 5. Add a public draw method that takes a GraphicsContextWrapper (you will need to import it from uk.ac.aston.oop.inheritance.util) and does not return anything (void): we will have subclasses override it. Since this is a generic shape, you cannot really draw it: just print a message with System.out.println() telling developers that you cannot really draw a Shape.

Make Rectangle extend Shape

Step 1. Update Rectangle so it extends Shape (meaning that all Rectangles are Shapes, so they should inherit everything from Shape):

public class Rectangle extends Shape {

Step 2. Remove all the fields and methods from Rectangle that Shape already has, except for draw which is different in this class.

Step 3. Change the constructor so it only calls the constructor of Shape (its super class) on its first line:

public Rectangle(double ulX, double ulY, double width, double height) {
  super(ulX, ulY, width, height);
}

Step 4. Add the @Override annotation to draw, so the Java compiler will ensure that it is really overriding something in Shape:

@Override
public void draw(GraphicsContextWrapper gc) {

Step 5. You will notice that draw still has a number of compilation issues, since upperLeftX and other fields are no longer available (they are private in Shape). Change the code so it uses the accessor methods instead: for instance, replace all mentions of upperLeftX with calls to getX().

Note that instead of rewriting draw to use the accessor methods in Shape, we could have made those fields protected in Shape. However, generally this is not recommended unless you have very strong reasons, as it goes against the principle of implementation hiding.

Step 6. Once the class compiles fine again, try renaming draw to drew for a moment: it should give an error message. What happens if you remove @Override while it's still called drew, and why do you think this happens? Return it to draw with the @Override annotation once you are done experimenting.

Make Ellipse extend Shape

We are going to follow a very similar approach for Ellipse.

Step 1. Make Ellipse extend Shape.

Step 2. Remove all the fields and methods of Ellipse that Shape already has, except for draw which is different in this class.

Step 3. Change the constructor so it only has one line which calls the constructor of the superclass, Shape:

public Ellipse(double upperLeftX, double upperLeftY, double width, double height) {
  super(upperLeftX, upperLeftY, width, height);
}

Step 4. Add the @Override annotation to the draw method, and change the code so it uses the get*() methods in Shape instead of the fields.

Make Circle extend Ellipse

We will do something similar to Circle, but it will extend Ellipse rather than Shape. This means we will inherit everything from both Shape and Ellipse. Follow these steps:

Step 1. Make Circle extend Ellipse.

Step 2. Change the constructor so it only calls its superclass constructor. The Shape constructor wants the upper left corner positions, the width, and the height of the shape: you will have to calculate those yourself in the arguments you give to super(...).

Step 3. Remove all the get* methods: the versions in Shape are enough, since in the constructor you already set the fields inherited from Shape.

Step 4. Remove the draw method: the version you inherited from Ellipse is enough, because it will call gc.oval() with the right values anyway.

Step 5. Change the getRadius() method so it computes it from getWidth() or getHeight(): that way you won't need radius anymore.

Step 6. Remove the centerX, centerY, and radius fields: you do not need these anymore.

In closing

  • You should by now have a new class Shape, which has helped reduce duplication across your other shape classes.
  • Rectangle and Ellipse should now extend Shape.
  • Circle should now extend Ellipse.
  • The program should run the same as before, and you should have no compilation issues.

4. Reusing overriden methods with super.method()

We want to create FilledRectangle, a variant of Rectangle that adds a fill to the rectangle of the user's choosing. To do it, we will reuse as much of Rectangle as possible.

Step 1. Create the FilledRectangle class, and have it extend Rectangle.

Step 2. Add a private field called fill of type javafx.scene.paint.Color (you can just type Color and have Eclipse help you import the right class).

Step 3. Create a constructor that takes a Color and the four usual arguments for a Rectangle (upper left corner X, upper left corner Y, width, and height). In its first line, it should call the super(...) constructor, and then it should set the fill field to the provided color.

Step 4. Add a public getFill method that returns the fill, in case some other class needs to know which fill was chosen.

Step 5. Now, write a draw method that @Overrides the one in Rectangle. It should use gc.fill to set the fill color, then use gc.fillRect to draw a filled rectangle, and finally use super.draw(gc) to call the version of draw in Rectangle which will draw the border.

Keep in mind that super refers here to the part of the current object (this) that is inherited from the Rectangle superclass, so with super.draw(gc) we mean "call the draw method inherited from Rectangle".

Step 6. Change the draw method in Launcher, replacing the line that creates a Rectangle with a line that creates a FilledRectangle (giving it Color.RED as its first argument). Try running the program: do you notice the difference?

Step 7. Add this at the beginning of the draw method of Rectangle:

gc.lineWidth(5);

Try running the program again. Do you see the difference?

Step 8. Change draw in FilledRectangle, moving the super.draw call to the top of the function. Try to run the program again: do you see the difference? This is an example of how the exact place where you call the inherited version of a method can be important.

5. Reuse by composition: Frame

You have seen how we can reuse code through inheritance, where we inherit fields and methods from a superclass, and we can override some methods with our own versions. We can also reuse the inherited versions with super.xyz(...) while we are overriding a method, as you saw in the previous section.

In this section, you are going to do what is called "reuse by composition". We are going to create Frame, a subclass of Shape that looks like a picture frame. Instead of implementing it fully ourselves, we are going to have Frame contain two Rectangles: to draw a Frame, we will just ask its two Rectangles to draw themselves.

Step 1. Create a Frame class that extends Shape.

Step 2. Add two protected fields of type Rectangle: one called outerRectangle, and one called innerRectangle.

Make sure you use these exact names and that they are protected - AutoFeedback will be looking out for that. We made them protected so AutoFeedback will be able to check you are using them correctly: the tests in AF belong to the same package as Frame, and protected allows access from subclasses and classes in the same package.

Step 3. Add a private static final int field called FRAME_THICKNESS: this will be the thickness of our frame. Set it to 10.

Step 4. Create a constructor that takes the usual four doubles: upper left X coordinate, upper left Y coordinate, width and height. It should follow these steps:

  1. Call the super(...) constructor.
  2. Set outerRectangle to a new Rectangle with the provided four doubles.
  3. Set innerRectangle to a new Rectangle where the upper left corner is FRAME_THICKNESS units down and to the right (add FRAME_THICKNESS to the upper left coordinates), and its width an height are decreased by FRAME_THICKNESS * 2.

Step 5. Create a draw method that overrides the one from Shape: it should simply ask both Rectangles to draw themselves.

Step 6. Change the draw method of Launcher, so it will create and draw a Frame instead. For instance, you could put the upper left corner at (200, 120), and give it 50 for the width and the height.

Step 7. Try running the application once more. You should be seeing the Frame and the FilledRectangle show up, like this:

Congratulations, you now know how to do reuse by composition! Simply put, it's implementing something new by combining (composing) things we already have.

6. Documenting with Javadoc

When developing a large system, in addition to writing the code, it is important to document what the different packages and classes are for, and how to best use them. Even if it is just you writing the code, you will appreciate having something that reminds you about how you are supposed to call that function you wrote a month ago!

Most modern programming languages have tools to write documentation, and Java has a tool called javadoc for it. In this section, you will describe all your shapes and your two main packages, and you will document in depth the Shape class (since it is the base class for all your shapes).

Describing our classes

First, we want to describe what our classes are. Add a Javadoc comment at the top of each class in the uk.ac.aston.oop.inheritance.shapes package saying what the class represents.

For instance, for Circle you might write something like this:

/**
 * Circle, specified as center + radius. 
 */
public class Circle ...

Don't just state the obvious: it's important that people know that Circle needs a center and a radius, and it may be important to let people know that Shape is specified through its upper left corner, width, and height.

Keep in mind that Javadoc comments use /** */, instead of /* */ like normal comments.

Describing our packages

Next, we want to describe what our packages are for. Each package should be a logical grouping of classes around the same idea. For instance, the uk.ac.aston.oop.inheritance package is the main package of the application and includes the main class, and the uk.ac.aston.oop.inheritance.shapes package contains all the geometrical shapes to be drawn.

If you want to describe a package with the javadoc tool, you need to create a package-info.java file inside the package, and add a Javadoc comment right above the package declaration.

Step 1. Right-click on the uk.ac.aston.oop.inheritance package in the "Package Explorer", and select "New - File". Enter package-info.java as the file name and click on "Finish".

Step 2. Write this into the file:

/**
 * Main package of the inheritance lab program.
 *
 * @author (your name)
 * @version 1.0
 */

package uk.ac.aston.oop.inheritance;

The first sentence gives a short summary of what the package is about.

The @author line says who wrote it.

The @version line mentions the version of the package ("1.0" is typically chosen for the first public release of a program).

Step 3. Do the same for the uk.ac.aston.oop.inheritance.shapes package: create the package-info.java file, and fill it in.

Describing Shape in depth

Shape is a very important class, because all our shapes inherit from it. We should document all its public methods and its constructor, so everyone knows how to use it.

Step 1. Add a Javadoc comment right above the constructor, like this:

/**
 * Creates a new instance.
 *
 * @param ulX X coordinate of the upper left corner.
 * @param ulY Y coordinate of the upper left corner.
 */
public Shape(double ulX, double ulY, double width, double height) {

Here, the first sentence summarizes what the constructor does, and then you have one @param name description line for each parameter. Add @param lines for width and height.

Step 2. Add a Javadoc comment right above the getX() method, like this:

/**
 * Returns the X coordinate of the upper left corner.
 *
 * @return X coordinate of upper left corner.
 */
public double getX() { return upperLeftX; }

This is the same approach, except here we have no @param lines (because we do not take any parameters). We do have a @return line explaining what the method returns.

Step 3. Add Javadoc comments to all the other public methods (all the get* methods and draw). Make sure to use @param and @return as needed.

Generating documentation in HTML format

Now that we have added these comments, we can use javadoc to generate documentation in a convenient web-based format. There are many ways to call the tool, but for this lab you will use a .launch file that we have prepared for you.

Step 1. Right-click on "Generate Javadocs (Inheritance Lab)" and select "Run As - Generate Javadocs". After some time, it should complete with BUILD SUCCESS: if it doesn't, you may have to fix your code or your Javadoc comments (check the "Console" tab for hints).

Step 2. In the "Package Explorer", open the target folder, then open site and apidocs, and then double click on index.html. You will see the generated documentation in HTML format.

Step 3. If you have done things correctly, the packages will have descriptions. If they do not show up, check your package-info.java files.

Step 4. Click on the uk.ac.aston.oop.inheritance.shapes package. If you have added Javadoc comments to each shape correctly, every class should have a description.

Step 5. Click on the Shape class. If you did it right, every method should have a description.

The idea is that you could then upload this apidocs folder to your website, and share that documentation with the other developers.

Java itself has Javadocs-based documentation, which you can check here. You should bookmark it for future reference!
